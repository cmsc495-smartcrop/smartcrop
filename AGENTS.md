# AGENTS.md

Guidance for AI coding agents working in this repository.

## Project overview

A smart crop monitoring platform. Field stations placed at farms send in
sensor readings (e.g. soil moisture, temperature, humidity) which the
platform ingests, stores, and surfaces to users.

- Backend: Go
- Frontend: Go `html/template` server-rendered UI (no SPA framework) using daisyUI, TailwindCSS and HTMX
- Database access: generated via `sqlc` from raw SQL — do not hand-write
  model structs or query code that sqlc is responsible for

## Repository structure

```
cmd/        # Entrypoints (main packages). One subdirectory per binary.
internal/   # Application code not intended for import outside this module.
ui/         # Go templates, static assets (CSS/JS/images) for the frontend.
db/         # SQL schema and migrations.
```

When adding new functionality, prefer extending `internal/` packages over
adding logic to `cmd/`. `cmd/` should stay thin — flag parsing, wiring,
and calling into `internal/`.

Current binaries: `cmd/web` (HTTP dashboard server), `cmd/seed` (DB
seeding), `cmd/broker` (embedded MQTT broker that ingests sensor readings
published to `<station-id>/<reading-type>` topics — see `internal/mqtt`
and `docs/mqtt-broker.md` for the topic/payload contract).

## sqlc workflow

- Queries live in `.sql` files (update the path below once confirmed) and
  are compiled by sqlc into Go code — **never edit generated files by
  hand**. Look for a `// Code generated by sqlc. DO NOT EDIT.` header.
- After changing a `.sql` query file or `sqlc.yaml`, regenerate with:
  ```
  sqlc generate
  ```
- Schema/migration changes go through migration files, not by editing
  generated models directly.

## Build, run, test

Fill in / correct these once confirmed — placeholders for now:

```
make serve/dev # start development server
go build ./...          # build all packages
go run ./cmd/<binary>    # run a specific service locally
go test ./...            # run all tests
go vet ./...             # static analysis
gofmt -l .               # check formatting (should output nothing)
```

Agents should run `gofmt`, `go vet`, and `go test ./...` before considering
a change complete.

## Conventions

- Follow standard Go idioms: small interfaces, explicit error handling
  (no swallowed errors), `errors.Is`/`errors.As` for error checks.
- Keep template files in `ui/` focused on presentation; business logic
  belongs in `internal/`.
- Station/reading domain terms should stay consistent across the codebase
  (e.g. pick one of "station" vs "sensor" vs "device" and use it
  everywhere — avoid mixing terms for the same concept).
- New DB access should go through sqlc-generated queries; if a query
  doesn't exist yet, add it to the relevant `.sql` file and regenerate
  rather than writing raw `database/sql` calls inline.

## What NOT to touch

- Generated sqlc output files.
- `go.sum` — let `go mod tidy` manage it, don't hand-edit.

## Notes for agents

- This file is a living document — if you notice it's out of date with
  the actual repo (wrong commands, missing directories, stale
  conventions), update it as part of your change.
